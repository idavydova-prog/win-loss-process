Project Purpose & Goal

Build a repeatable Win/Loss story library for the AMER PACE RCG team — covering both attrition (losses) and full renewals with zero attrition (wins) — that can be used during Monthly Business Reviews (MBRs) and presented to leadership (Rick, Kathleen, Colin).

What Success Looks Like
  • Individual stories: Each closed renewal ≥ $100K attrition has a structured narrative: Situation → Response → Outcome → Learnings, with hashtag tagging.
  • Portfolio view: Leadership can see total attrition by segment (e.g., Commerce total, Marketing total), not just individual accounts.
  • Win view: Renewals closed with zero attrition are equally tracked — a complete picture of the quarter.
  • Program effectiveness: Hashtag tagging (#SWAP, #ADOPT, #RFP, etc.) allows themes to surface — which plays saved accounts, which were missed.
  • Scalable & low-CSM-burden: Data entry owned by Program Leads; CSMs are input sources only for context/enrichment.
Current Status

Manual process is paused for validation. Ilona has built the Commerce sheet and sample stories. Brandi must validate the format and questions before any broader rollout or leadership presentation.

Story Format (Established)

Stories are structured in four sections:

  • Situation — What was happening at the account going into renewal
  • Response — What the team did to mitigate or save it
  • Outcome — What happened (dollar impact, logo retained/lost)
  • Learnings — What would have changed the result; hashtag tagging

Current Process (As-Is)

How a Story Gets Built Today
1

Filter renewal records — Pull renewal records for the relevant team (Commerce, Marketing, etc.) where Attrition Amount ≥ $100K. Source: SF Report export into Google Sheet.

Owner: Ilona / Program Leads

2

Construct opportunity hyperlink — Because SF Report only surfaces Account-level links, a manual Excel formula is used to construct the Renewal Opportunity URL from the Opportunity ID. This is fragile and manual.

Current workaround: ="https://org62.lightning.force.com/lightning/r/Opportunity/"&[ID cell]&"/view"

3

Enrich with CSM context — Lead OUs or CSMs manually add qualitative answers: what program ran, whether logo was retained, Signature status, counterfactual. This supplements what's in SF Manager Notes.

Owner: Program Leads (NOT individual CSMs — confirmed decision)

4

Write the story — Using the Situation/Response/Outcome/Learnings framework, a story narrative is drafted per account.

Owner: Ilona (currently); intended to scale to PLs

5

Apply hashtags — Tag the story with the applicable program hashtags (#SWAP, #ADOPT, #RFP, #IMPLEMENT, etc.) and flag any discrepancies vs. what was tagged in the SF Red Account record.

Needs audit process via Slack skills — not yet built

6

Aggregate into portfolio view — Roll up individual stories into a segment-level summary (total attrition, total wins, themes by reason).

Not yet built — identified as a gap by Brandi

What Triggers the Process

Confirmed trigger: A renewal record closes at or above a defined dollar threshold — regardless of whether a Red Account exists. This ensures the ~20% of attrition that never gets a Red Account record is still captured.

Threshold: Currently $100K — but the exact number has not been formally locked by Brandi. Needs confirmation.

Data Discrepancies & Gaps

Discrepancy 1 — Same Account, Different Attrition Reason

Under Armour, Inc. (Account ID: 0013000000FIFKI) appears in both the All RCG sheet and the Commerce sheet with a different Attrition Reason:

  • All RCG: Attrition Reason = "Concessions"
  • Commerce: Attrition Reason = "Customer Relationship"

Both show the same sub-reason ("Product-Market Fit Mismatch/Oversold") and same attrition amount ($2.29M). This is either a SF data entry inconsistency or the Commerce sheet was built from a different report pull.

Discrepancy 2 — Inconsistent Manual Column Names

The same qualitative concepts are named differently across sheets, making consolidation or automated roll-up impossible without manual remapping. Key problem areas:

  • "Offense Program (Hashtags)" vs. "What did we do to try to save it" — structured tag vs. free text
  • "Amount AOV Retained?" vs. "Amount (retained)" vs. "Renewed Amount" — unclear if these measure the same thing
  • "Success Plan" (Core Non-Digital) vs. "Did the customer drop Signature support?" (Commerce/Marketing) — different question entirely
Discrepancy 3 — Wins Sheet Manual Fields Are Largely Unpopulated

The RCG - All - Wins sheet has the manual enrichment columns but they are blank for nearly all records. Wins are not being tracked with the same rigor as losses.

Discrepancy 4 — Renewal Opportunity Link Is Fragile

Commerce and Marketing sheets use an Excel formula to construct the Renewal Opportunity URL. This breaks when exported to CSV. The Wins sheet has an 18-char ID (RENEWAL_OPTY_ID_18) that is natively link-ready — that approach should be standardized across all sheets.

Discrepancy 5 — Duplicate "Full/Partial" Column in Commerce Sheet

The Commerce sheet has "Full/Partial" as both column D (SF export) and column N (manual entry). These may diverge over time.

Discrepancy 6 — Unnamed Columns in All RCG Sheet

The All RCG sheet has 13 named columns followed by unnamed (None) columns. Unknown what data is in those columns. Needs inspection.

Discrepancy 7 — $100K Threshold Not Applied Consistently

The Commerce sheet is filtered to ≥$100K. The All RCG, Core Non-Digital, and Marketing sheets contain all records with no threshold filter. There is no "in scope" flag to indicate which records qualify for Win/Loss stories.

Discrepancy 8 — Multiple Tracking Spreadsheet Versions

Brandi has a separate tracking version not yet merged into the shared team template. Risk of parallel data entry and duplicate records until resolved.

Discrepancy 9 — Contact POC Not Captured in All Sheets

Commerce and Marketing sheets have "Contact POC" (e.g., "Steve Larocca/Ilona Davydova"). All RCG and Core Non-Digital do not. This will be lost if data is merged without the field.

Discrepancy 10 — No "In Scope" Flag or Threshold Filter Field

No dedicated column marks a record as "in scope for Win/Loss story." PLs must manually filter each time, creating inconsistency in which records get processed.

What Has to Be True Before This View Exists

The portfolio view has hard dependencies. None of these are optional — skip any one of them and the rollup either doesn't work or produces numbers that can't be defended in front of leadership.

!

Q2 (Program/Hashtag) must be a structured picklist, not free text

Free-text answers in Commerce and Marketing cannot be counted or grouped. "We ran a SWAP motion" and "#SWAP" are the same thing to a human but different strings to a formula. The program effectiveness layer only works with structured tags.

!

Q4 (Logo/License Outcome) must use consistent values across all sheets

"What Lost: Logo, License or Signature" and "Did we lose the logo, retain it, or grow it?" produce incompatible values. The Logos Lost / Retained / Grown columns in the scorecard only work if every segment uses the same answer options.

!

Wins must be tracked in the same sheet structure as losses

The Wins sheet has a different schema. Until wins and losses share the same column names, the "Wins ($)" column in the portfolio scorecard cannot be built from a single formula — it requires separate lookups against a differently structured dataset.

~

A "Segment" tag must exist on every record

Currently each segment lives in its own sheet. For a single portfolio summary tab to roll up across Commerce + Marketing + Core Non-Digital, each row needs a segment identifier. This can be derived from SALES_OU_1 or added as a static column — either works, but it must be present and consistent.

~

FEB1_ATTRITION interpretation must be confirmed before wins are counted

The Wins ($) column depends on correctly identifying zero-attrition renewals vs. attrition-improvement recoveries. Until the FEB1_ATTRITION field interpretation is confirmed by the reporting team, any wins dollar total carries a risk of double-counting or under-counting recoveries.
